# Quickstart: npmdata v2

**Date**: 2026-03-08  
**Branch**: `001-v2-rewrite`

---

## Overview

npmdata v2 lives in `lib/src/v2/` — a new subfolder within the existing `lib/` project. It is a **clean-room rewrite** that reuses the `lib/` toolchain (tsconfig, jest, eslint, Makefile). No existing file in `lib/src/` (v1 code) or outside `lib/` is touched.

---

## Prerequisites

- Node.js LTS (20.x)
- pnpm 8.x: `npm install -g pnpm`

---

## Bootstrap the v2 Project

```bash
cd lib
make build
make lint-fix
make test
```

No separate `pnpm install` is needed — v2 shares `lib/`'s existing dependencies.
All three commands should succeed with no errors before any further work.

---

## File Layout

```
lib/src/v2/
├── cli/          # UI layer: argument parsing, console output, config loading
├── package/      # Orchestration: action coordination, config merging, symlinks, replacements
└── fileset/      # File layer: diff computation, execute, markers, gitignore, package install
```

Compiled output lands in `lib/dist/v2/` (generated by `make build` in `lib/`).

---

## Running the CLI Locally

After `make build` in `lib/`:

```bash
# Extract from a package
node lib/dist/v2/main.js extract --packages my-pkg@^1.0.0 --output ./output

# Check sync status
node lib/dist/v2/main.js check --output ./output

# List managed files
node lib/dist/v2/main.js list --output ./output

# Purge managed files
node lib/dist/v2/main.js purge --output ./output

# Init a new publishable data package
node lib/dist/v2/main.js init --output ./my-new-package
```

## Using a Config File

Create `.npmdatarc` (JSON) in your project root:

```json
{
  "sets": [
    {
      "package": "my-org-kit@^2.0.0",
      "output": {
        "path": "./data"
      },
      "selector": {
        "files": ["docs/**", "*.md"],
        "presets": ["basic"]
      }
    }
  ]
}
```

Then run:

```bash
node lib/dist/v2/main.js extract --presets basic
```

## Using a Self-Installable Package

If you have a data package `my-org-kit` that was scaffolded with `init`:

```bash
# Consumer installs and runs the data package directly
npx my-org-kit
npx my-org-kit --presets extended --dry-run
```

The package's `bin/npmdata.js` calls `run(__dirname)` from v2's library, reads `npmdata.sets` from its own `package.json`, and applies the same extraction logic.

## Architecture Quick Reference

```
CLI invocation
   └─ cli/cli.ts          (cosmiconfig load, command dispatch)
         └─ cli/commands/extract.ts   (argv parse)
               └─ package/action-extract.ts  (fileset loop, rollback tracking)
                     ├─ fileset/diff.ts      (Phase 1: pure read-only diff → ExtractionMap)
                     │      └─ [if dependency package] package/action-extract.ts  (recursive)
                     └─ fileset/execute.ts   (Phase 2: disk writes, marker, gitignore)
                           └─ package/symlinks.ts + content-replacements.ts  (post-extract)
```

**Key invariant**: `fileset/diff.ts` never writes to disk. All writes happen in `fileset/execute.ts`. This separation enables `check`, `list`, and `purge` to reuse the diff result and enables independent unit testing of each phase.

## Key Differences from v1

| Aspect | v1 (`lib/src/`) | v2 (`lib/src/v2/`) |
|--------|-------------|------------|
| Execution model | Subprocess per entry (node CLI) | Direct in-process function calls |
| Config schema | `presets`, `upgrade` at root level | Nested in `selector.presets`, `selector.upgrade` |
| Two-phase model | Single-pass per subprocess | Explicit diff phase then execute phase |
| Rollback | None | Newly created files deleted on failure |
| check drift detection | Marker-only | Full filtered package source vs output dir |
| check `--presets` | Not supported | Supported |
| purge symlink cleanup | Via subprocess | Direct call to symlink cleanup |
| Backward compatibility | n/a | Intentionally broken from v1 |
